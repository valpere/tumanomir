## Ключові рішення, вже ухвалені (з підставами)

1. **D_pair — робоча метрика, не ентропія.** Консенсус 5/5 рецензентів +
   власний експеримент: H@0.95 сатурує на log₂N для реальних документів
   (усі real-спеки дали 3.32 біт при N=10), дискримінує лише середня
   попарна відстань.
2. **Теорія через умовну ентропію H(C|S,θ)**, НЕ через суму дисперсій
   σ²spec+σ²model — останнє рецензенти 4/5 розібрали як категоріальну
   помилку (несепарабельні компоненти, ентропія ≠ дисперсія).
3. **K_drift strict-режим — фундамент**: детермінований лінтер розмітки,
   без LLM, найзахищеніша метрика перед senior-аудиторією. Assisted
   (LLM-парсер графа) — roadmap, успадковує похибку екстракції.
4. **D_const v0.1 — чесний lexical proxy**, не графова метрика. Повний
   RFLP-граф (Neo4j) — roadmap. Формула щільності карає прозу без
   обмежень, не прозу взагалі.
5. **Baseline зі статті — copy-floor** (транскрипція готових типів дала
   H=0 на обох приладах), не універсальна підлога. Baseline-suite —
   roadmap для `calibrate`.

sim на двох незалежних приладах.

## Граблі Ollama (з експерименту, вбудовано в requirements)

- thinking-режим з'їдає num_predict на складних спеках → порожній content;
  збій корелює зі складністю входу → систематичне спотворення. `think:false`.
- дефолтний num_ctx (4096) мовчки обрізає великі промпти.
- num_predict нижче природної довжини → тронкейшн → фальшивий «окремий
  кластер».
- Хмарні `:cloud` моделі ~6× швидші за локальні 30b.
