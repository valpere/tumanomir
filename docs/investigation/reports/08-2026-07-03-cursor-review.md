# Рецензія плану «Джерело Невідомості»

Незалежна оцінка на основі `SourceOfTheUnknown-Plan.md` та протоколу/результатів у `sanity/README.md`.

---

## 1. Сильні сторони

**Теза та діагностика проблеми.** Центральна ідея — специфікація як розподіл інтерпретацій з явним «бюджетом точності», а не як бінарне джерело істини — добре аргументована й своєчасна для контексту ШІ-агентів. Розбиття «туману» на неоднозначність, неповноту й нетрасованість дає зрозумілу онтологію; прив’язка кожного режиму до окремої метрики — сильний інженерний хід.

**Чесність щодо меж.** Розділ «Межі підходу» (instrument-relative вимір, низька ентропія ≠ правильність, геймінг метрик, пороги як гіпотези, зона пошуку vs реалізації) — рідкісна зрілість для такого матеріалу. Це саме те, що знімає частину заперечень senior-аудиторії.

**Практичний sanity check.** Експеримент не залишився декларацією: два незалежні прилади, відтворюваний пайплайн (`gen.sh` / `analyze`), фіксація граблів (think-режим, `num_ctx`, `num_predict`, discard count). Висновок про `Decimal(10,2)` → різні Go-мапінги при $H_{sharp}=1.77$ — переконливий приклад «метрика вказує на залишковий люфт», а не маркетинг.

**Архітектурний кейс.** Каскад Router → Coder → DB → context accumulation — конкретний, перевірюваний наратив про те, як низька щільність обмежень масштабується в конвеєрі. Mermaid-діаграма й контраст fog/sharp добре ілюструють механізм.

**$K_{drift}$ як детермінований шар.** Відсутність LLM у контурі вимірювання для трасованості — правильний вибір для CI/git hook; формула проста й зрозуміла.

**Розділення режимів дисперсії.** «Зона реалізації» vs «зона пошуку» з протилежними політиками оркестратора — важлива концептуальна поправка до наївного «ентропія = зло».

## 2. Слабкі місця та технічні помилки

### Термінологія та математика

**«Дисперсія» vs ентропія.** $H_{spec}$ — це ентропія Шеннона розподілу кластерів генерацій, а не дисперсія у статистичному сенсі. Формула

$$\sigma^2_{\text{вихід}} = \sigma^2_{\text{специфікація}} + \sigma^2_{\text{модель/семплінг}}$$

і подальше $\Delta H = H_{spec} - H_{baseline}$ змішують дві різні величини (дисперсію та ентропію). Для senior-аудиторії це виглядає як категоріальна помилка, навіть якщо інтуїція «шум + сигнал» правильна. Потрібно або перейменувати метрику (наприклад, $H_{spec}$ / «ентропія інтерпретацій»), або формально визначити дисперсію на тому ж просторі AST-відстаней і показати зв’язок з $H$.

**Твердження $H_{fog} \gg H_{sharp} > H_{baseline}$ не тримається на приладі B при порозі 0.95.** За таблицею: B дає $H_{sharp}=3.32$ і $H_{fog}=3.32$ — сатурація стелі. Ранжування за $H$ зламалось; воно відновлюється лише при $H_{@0.80}=2.12$ для sharp. У тексті це згадано як нюанс, але головний висновок sanity check формулюється так, ніби порядок стабільний — це слабке місце перед критичним читачем.

**«Шумова підлога нульова» — переузагальнення.** `baseline.md` буквально містить готові Go-типи з інструкцією «відтвори рівно це». $H=0$ означає, що модель копіює заданий шаблон, а не що прилад загалом безшумний. При temp=1.0 нульова ентропія на copy-task не доводить $\sigma^2_{\text{модель}}=0$ для довільної формалізованої специфікації. Висновок «виміряна деінде дисперсія — властивість документа, а не семплінгу» занадто сильний.

**$N=10$, temp=1.0 як основний режим.** У плані temp≈1.0 названо стрес-тестом, а temp≈0.3–0.5 — робочим вимірюванням; експеримент і таблиці побудовані на 1.0/10. Це максимізує шум приладу саме тоді, коли теза про baseline-корекцію ще не працює (baseline=0). Внутрішня суперечність між протоколом і інтерпретацією.

### Метрики $D_{const}$ та $K_{drift}$

**$D_{const}$ — евристика без валідації.** Формула (типізовані ребра / неструктуровані текстові вузли) не має експериментального підтвердження в sanity check. Знаменник «неструктуровані текстові вузли» погано операціоналізований: rationale, приклади, посилання на ADR — все карається однаково. Застереження в тексті є, але поріг 0.35 у YAML-gate виглядає так само «магічно», як і 0.2 для $K_{drift}$.

**$K_{drift}$ залежить від якості парсингу.** «Легкий AI-агент-парсер» vs лінтер — це два різні прилади з різним bias. Без узгодженого екстрактора графа метрика не відтворювана між командами.

### Операціоналізація $H_{spec}$

**AST bag-of-features + cosine — грубий проксі семантики.** `analyze/main.go` рахує частоти рядків на кшталт `field:Transaction:Amount:int64`; перейменування типів або reorder полів змінює вектор без зміни семантики. Два еквівалентні дизайни можуть потрапити в різні кластери — метрика завищує ентропію.

**Single-linkage @ фіксованому порозі.** Відомий chaining effect; поріг 0.95 — довільна частина приладу (для kimi критично). Без sensitivity analysis по порогу senior читач поставить під сумнів усі абсолютні значення.

**Задача генерації.** «Go type definitions» — один канал інтерпретації. Інша постановка (OpenAPI, SQL DDL, TLA+) дасть інший $H$. Це не помилка, але має бути явно названо operationalization, а не «неоднозначність документа» взагалі.

### Риторичні твердження без опори

**«$X \times 10$» мультиплікатор ШІ** — метафора без визначення $X$ і без емпірики; для технічної статті краще прибрати або замінити посиланням на вимірюваний ефект (токени, LOC, кількість агент-ітерацій).

**Пороги в `quality_gates` (0.75, 0.35, 0.2)** — названі гіпотезами в одному місці й подані як готовий YAML в іншому. Без калібрувального корпусу це виглядає як псевдо-константи.

**Реальні документи та сатурація.** Усі `real-*` при $H_{@0.95}=3.32$ — метрика втрачає дискримінацію; план це визнає, але таблиця все одно центрує $H$, хоча єдиний стабільний сигнал там — `meanPairwiseSim`. Твердження про інверсію ragivka vs session-indexer спирається на одну пару чисел (0.282 vs 0.524 на A; близькі 0.278 vs 0.258 на B) без довірчих інтервалів.

**Контрольний `real-ragivka-full`.** Один datapoint (+41% sim на B); прилад A не прогнаний через обмеження контексту — асиметрія порівнянь.

## 3. Що покращити

1. **Уніфікувати термінологію:** «ентропія інтерпретацій» замість «дисперсія» там, де рахується Shannon $H$; прибрати або переформулювати variance-decomposition.

2. **Переписати висновки sanity check під фактичні дані:** на B ранжування за $H_{@0.95}$ не працює; головний робочий сигнал при $N=10$ — `meanPairwiseSim` і/або $H$ при каліброваному порозі per-instrument.

3. **Перевизначити baseline:** окремо «copy-task floor» (еталон з готовим кодом) і «мінімально формалізована спека без готових типів» — щоб відділити шум моделі від властивостей документа.

4. **Додати sensitivity table:** пороги 0.80/0.90/0.95, temp 0.3/0.5/1.0 — хоча б для fog/sharp/baseline; показати, які висновки стійкі.

5. **Статистика на малій вибірці:** bootstrap по перегенераціях або хоча б Wilson/байєсівські кредибільні інтервали для частки кластерів; інакше різниця 0.278 vs 0.282 статистично нічого не означає.

6. **Miller–Madow або заміна метрики:** якщо згадано в тексті — реалізувати в `analyze`, або чесно позиціонувати $H$ лише як ordinal gate («1 vs 10 кластерів»).

7. **Формалізувати $D_{const}$ і $K_{drift}$:** JSON-схема вузлів/ребер, приклад графа для sharp/fog, один скріншот/лічильник — інакше дві з трьох метрик лишаються paperware.

8. **Прибрати або пом’якшити YAML-gates** до появи калібрування; залишити як illustrative pseudoconfig.

9. **Розділити narrative (Акти I–III) і spec (Акт V):** senior-читачі часто хочуть формальні визначення в перших 2–3 сторінках, а не після 50 рядків риторики.

10. **Узгодити твердження «порівнювати між моделями безглуздо»** з використанням двох приладів як валідації: коректна формулювання — «ранжування ordinal може збігатися; абсолютна шкала — ні».

## 4. Що додати

| Блок | Навіщо |
|------|--------|
| **Формальне визначення «бюджету точності»** | Вектор $(K_{drift}, D_{const}, H_{spec})$ + зона (impl/exp) + допустима область; зараз це розмито |
| **Зв’язок з існуючими практиками** | IEEE 29148, OpenAPI lint, Semgrep/ArchUnit, mutation testing — де ваш підхід доповнює, а не дублює |
| **Негативні класи збоїв** | Суперечливі вимоги, застарілість, контекстні залежності — зараз три режими не покривають «конфлікт у тексті» |
| **Human baseline** | 2–3 інженери незалежно інтерпретують fog/sharp → порівняння з LLM-$H$ (concurrent validity) |
| **Альтернативна операціоналізація** | Той самий fog/sharp → OpenAPI або DDL → чи зберігається порядок |
| **Failure analysis по кластерах** | Приклад diff між кластерами fog (які саме рішення розходяться) — перетворює метрику на actionable feedback |
| **Вартість вимірювання** | Токени/час на один прогін $H_{spec}$; коли дешевший лексичний $D_{const}$ достатній |
| **Зв’язок з downstream quality** | Чи корелює $H_{fog}$ з реальною кількістю compile-fix ітерацій у конвеєрі — без цього метрика лишається input-only |
| **Обмеження AST-метрики** | Приклад семантично еквівалентних, AST-різних генерацій |
| **Повна таблиця reproducibility** | Версії Ollama, commit hash моделей, `num_ctx`/`num_predict`, системний промпт (зараз частково розкидано) |

## Підсумок для senior-аудиторії

План сильний як **концептуальний каркас** і як **перший відтворюваний прототип** $H_{spec}$ з чесними застереженнями. Слабкість — у **захищеності кількісних тверджень**: змішання ентропії з дисперсією, сатурація $H$ при $N=10$, залежність від порогу кластеризації, нульовий baseline на copy-task і відсутність валідації для $D_{const}$/$K_{drift}$.

Рекомендація перед публікацією: знизити градус категоричності в sanity check («підтвердило гіпотезу на toy+corpus з обмеженнями»), винести `meanPairwiseSim` і sensitivity analysis на рівень з $H$, формалізувати дві «сухі» метрики або чітко позначити їх як roadmap. Після цього матеріал витримає дискусію з архітекторами та ML-інженерами; у поточному вигляді найуразливіші місця — математична термінологія, твердження про нульовий шум приладу та стабільність ранжування між приладами.
